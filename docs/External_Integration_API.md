# دليل التكامل مع الأنظمة الخارجية (External Systems Integration Guide)

## 1. مقدمة (Introduction)
يوفر هذا النظام واجهة برمجية (API) مبسطة وصممت خصيصاً لتمكين الأنظمة الخارجية (مثل الأنظمة البنكية، تطبيقات الويب، أو CRM) من إرسال رسائل WhatsApp بكل سلاسة. 

**فلسفة التصميم:**
- **البساطة**: لا يحتاج النظام الخارجي لمعرفة تفاصيل المزود (Vonage) أو تعقيدات القوالب. فقط "إلى من" و "ما هي الرسالة".
- **الموثوقية**: الاعتماد على Vonage كخيار افتراضي (Default) مع آليات معالجة الأخطاء.
- **التوثيق**: تخزين كامل لكافة العمليات في ملفات السجلات (Logs) لضمان الأمان والمراجعة.

---

## 2. نقطة الاتصال (API Endpoint)

يتم إرسال جميع الطلبات إلى الرابط التالي:

| الطريقة (Method) | المسار (Path) |
| :--- | :--- |
| **POST** | `/api/whatsapp/send` |

---

## 3. صيغة الطلب (Request Format)

يقبل النظام طلبات بصيغة JSON. تم تبسيط الحقول المطلوبة إلى الحد الأدنى لتسهيل الربط.

### الحقول المطلوبة (Required Fields):

| الحقل (Field) | النوع (Type) | الوصف (Description) | أمثلة (Example) |
|---|---|---|---|
| **`to`** | `string` | رقم هاتف المستقبل. يفضل استخدام الصيغة الدولية (بدون + أو 00، أو معهم، النظام يعالجها). | `"971500000000"` <br> `"14157386102"` |
| **`message`** | `string` | نص الرسالة المراد إرسالها. <br> *(ملاحظة: يمكن استخدام `body` بدلاً من `message` if desired)* | `"مرحباً، رمز التحقق الخاص بك هو 1234"` |

### مثال على الطلب (cURL Example):
```bash
curl -X POST http://YOUR-DOMAIN/api/whatsapp/send \
  -H "Content-Type: application/json" \
  -d '{
    "to": "971501234567",
    "message": "شكراً لاستخدامك خدماتنا. تم استلام طلبك بنجاح."
  }'
```

### حقول اختيارية (Optional):
يمكنك إرسال حقول إضافية للتتبع (Meta Data) إذا رغبت، لكنها **غير إلزامية**.

```json
{
  "to": "971501234567",
  "message": "Test Message",
  "meta": {
    "txnId": "REF-12345",
    "sourceSystem": "CoreBanking"
  }
}
```

---

## 4. آلية العمل (System Workflow)

عند استلام الطلب، يقوم النظام بالخطوات التالية تلقائياً:

1. **التحقق (Validation)**:
   - التأكد من وجود `to` و `message`.
   - إذا تم استخدام `message`، يقوم النظام بتحويلها داخلياً لمعالجتها.

2. **التوجيه (Routing)**:
   - يستخدم النظام مزود **Vonage** بشكل افتراضي لإرسال الرسالة.
   - في حال كان النص إنجليزياً أو عربياً، يتم التعامل معه بترميز UTF-8 المناسب.

3. **التسجيل (Logging)**:
   - يتم حفظ نسخة من الطلب والرد في ملفات السجلات في المجلد `logs/messages-YYYY-MM-DD.log`.
   - هذا يضمن وجود أرشيف كامل لكل رسالة خرجت من النظام، مستقلاً عن قاعدة البيانات.

4. **الرد (Response)**:
   - يعيد النظام حالاً نتيجة الإرسال للنظام الخارجي.

---

## 5. صيغة الرد (Response Format)

يرد النظام بـ JSON يوضح حالة العملية.

### أ. في حالة النجاح (Success - 200 OK):
```json
{
  "success": true,
  "provider": "vonage",
  "providerMessageId": "dbe5aca8-...",
  "rawResponse": {
    "message_uuid": "dbe5aca8-..."
  }
}
```
* **`success`**: `true` تعني أن Vonage استقبل الطلب.
* **`providerMessageId`**: رقم تتبع الرسالة من Vonage.

### ب. في حالة الخطأ (Error - 400/500):
```json
{
  "success": false,
  "errorCode": "VALIDATION_ERROR",
  "errors": [
    "Recipient phone number (to) is required."
  ]
}
```

---

## 6. التعامل مع الأخطاء الشائعة

| الخطأ | السبب المحتمل | الحل |
|---|---|---|
| `VALIDATION_ERROR` | لم يتم إرسال `to` أو `message`. | تأكد من صيغة JSON واسماء الحقول. |
| `INVALID_JSON` | تنسيق JSON غير صالح. | تأكد من إغلاق الأقواس والفواصل. |
| `PROVIDER_ERROR` | خطأ من جانب Vonage (مثلاً الرقم غير صحيح). | راجع `rawResponse` في الرد للتفاصيل. |

## الخلاصة
تم تصميم هذا الربط ليكون "ركب وشغل" (Plug and Play). أرسل `to` و `message`، وسيقوم النظام بالباقي.
